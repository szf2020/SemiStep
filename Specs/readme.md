# Техническое задание: Приложение для редактирования и исполнения рецептов

## 1. Общие сведения

### 1.1. Назначение системы
Приложение предназначено для:
- создания, редактирования и проверки технологических рецептов (последовательностей шагов);
- передачи рецептов в ПЛК и обратного чтения;
- контроля состояния исполнения рецепта в режиме онлайн;
- сохранение файла рецепта в файл и чтение из файла;
- взаимодействия с внешними системами (SCADA) по OPC для обработки событий (?);

Цель — предоставить удобный инструмент работы с рецептами, независимый от конкретной SCADA-системы.

### 1.2. Область применения
- ПК на Windows (Linux (?)) подключенный к Mitsubishi/Siemens PLC по Modbus TCP.
- Работа инженеров, технологов и операторов на рабочих местах под управлением ОС семейства Windows.

### 1.3. Целевая аудитория
- Технологи – разработка и корректировка рецептов.
- Инженеры – настройка конфигурации, интеграция с ПЛК, диагностика.
- Операторы – запуск и мониторинг исполнения рецептов (с ограниченными правами редактирования).

## 2. Требования к пользовательскому интерфейсу

### 2.1. Оконная модель

#### 2.1.1. Окно редактирования/исполнения рецепта
- Есть основное окно работы с рецептом, позволяющее:
  - редактирование рецепта;
  - запуск и мониторинг исполнения рецепта.
- В этом окне отображаются:
  - текущий активный рецепт;
  - состояние исполнения (если есть связь с ПЛК/OPC);
  - основные команды: загрузка/выгрузка в ПЛК, запуск/останов, переключение режимов отображения.

#### 2.1.2. Окна исключительно для редактирования
- Приложение должно поддерживать **произвольное количество** дополнительных окон, предназначенных:
  - только для редактирования рецептов;
  - без управления исполнением и загрузки в плк.
- В этих окнах:
  - недоступны команды запуска/остановки исполнения;
  - допустимы операции по редактированию, копированию, сохранению/загрузке рецептов с диска.

### 2.2. Общий вид окон
В каждом окне редактирования (и в основном окне исполнения):
- основная область с табличным представлением рецепта;
- панель инструментов с основными командами для данного типа окна;
- статус-бар с индикацией подключения к ПЛК/OPC и состояния синхронизации;
- панель сообщений/логов.

### 2.3. Табличное представление рецепта

#### 2.3.1. Базовый режим (классический)
- Строки соответствуют шагам рецепта (1..N).
- Столбцы соответствуют параметрам и служебным полям шага.
- Заголовок столбцов отображает названия параметров.
- Поддерживаются до ~1000 шагов и до ~30 параметров.

#### 2.3.2. Транспонированный режим
- Заголовки параметров отображаются **слева** (каждый параметр — отдельная строка).
- Сверху, слева направо, отображаются **номера шагов** (1..N).
- Данные рецепта в обоих режимах опираются на единую модель.
- Переключение между режимами осуществляется по установке параметра в настройках.

### 2.4. Редактирование и навигация
- Редактирование ячеек по аналогии с табличными редакторами.
- Базовый набор горячих клавиш:
  - `Ctrl+C`, `Ctrl+V`, `Ctrl+X` — копирование/вставка/вырезание;
  - `Ctrl+N`, `Del` — добавление и удаление шагов;
  - `Ctrl+Z`, `Ctrl+Y` — Undo/Redo.
- Визуальная индикация:
  - выделение активной строки/ячейки;
  - подсветка текущего шага исполнения (если есть данные от ПЛК/OPC).

### 2.5. Блокировка редактирования
- При активном исполнении рецепта (по данным от ПЛК/OPC или внутреннему состоянию):
  - редактирование может быть полностью или частично заблокировано;
  - недоступны операции, которые могут привести к неконсистентности исполняемого рецепта (удаление/добавление шагов, изменение критичных параметров).

## 3. Модель данных и рецепта

### 3.1. Структура рецепта
- Рецепт: упорядоченный список шагов.
- Каждый шаг:
  - тип действия;
  - набор параметров (значения по параметрам, определённым конфигурацией);
  - комментарий и служебные поля (при необходимости).

### 3.2. Ограничения
- Количество шагов: до ~1000.
- Количество параметров: до ~30.
- Конкретный набор параметров и их типы задаются конфигурацией, получаемой из ПЛК.

### 3.3. Валидация данных
- На основе типов параметров и их ограничений (диапазоны, формат, обязательность).
- При вводе недопустимых значений:
  - отображается понятное сообщение;
  - запрещается сохранение/отправка рецепта в ПЛК до исправления ошибок.

---

## 4. Конфигурация, получаемая из ПЛК

### 4.1. Общая концепция
- Конфигурация структуры рецепта и таблицы полностью хранится в ПЛК.
- Обмен конфигурацией выполняется по Modbus TCP.
- На стороне ПЛК выделяется область «header», содержащая служебную информацию, и связанные с ней области, описывающие структуру таблицы и колонок.

### 4.2. Header-конфигурации (область служебной информации)
В header должны храниться, по крайней мере, следующие данные (конкретный формат и адреса уточняются):

- Признаки формата и представления:
  - порядок байтов для чисел с плавающей точкой (little-endian / big-endian).
- Информация о состоянии и версии:
  - текущий **hash** или другая контрольная сумма конфигурации;
  - признак `processing` (занятость/идёт обработка на стороне ПЛК);
  - признак `recipe_invalid` (неконсистентность текущего рецепта по мнению ПЛК).
- Размеры и адреса областей:
  - количество шагов рецепта (максимум и/или текущее значение);
  - размер и базовый адрес области INT;
  - размер и базовый адрес области FLOAT;
  - (при необходимости) адреса областей, в которых хранятся описания колонок, имена и прочие метаданные.
- Иная служебная информация:
  - флаги/версии, необходимые для согласования формата конфигурации между приложением и ПЛК.

(?) Возможно, потребуется хранить в области памяти MagicNumber (произвольное число), для подтверждения подключения ПЛК.

### 4.3. Описание колонок и таблицы
- В ПЛК должны храниться данные, описывающие:
  - количество колонок;
  - для каждой колонки:
    - идентификатор/ключ;
    - отображаемое имя (для заголовка);
    - тип данных (int, float, string);
    - признаки редактируемости (read-only / read-write);
    - привязку к области памяти (INT/FLOAT, смещение для конкретного шага);
    - (при необходимости) параметры отображения (например, формат числа).
- Приложение при старте:
  - читает header;
  - по информации header находит и читает описание колонок и структуры таблицы;
  - строит внутреннюю модель конфигурации, на основании которой:
    - формирует таблицу UI;
    - знает, как маппить значения шагов на регистры ПЛК.

### 4.4. Загрузка конфигурации
- Конфигурация считывается из ПЛК при запуске приложения.
- При ошибке чтения конфигурации:
  - приложение уведомляет пользователя;
  - блокирует операции с рецептами, зависящие от конфигурации, до успешного чтения.

## 5. Взаимодействие с ПЛК

### 5.1. Протокол и транспорт
- Основной протокол: Modbus TCP.
- Настройки подключения:
  - IP-адрес;
  - порт;
  - UnitId (если нужен);
  - таймауты и количество попыток.

### 5.2. Области памяти
- Область header (служебная информация о конфигурации и состоянии).
- Области:
  - конфигурации колонок и структуры таблицы;
  - данных рецепта (INT/FLOAT/STRING);
  - служебных регистров состояния исполнения рецепта (номер шага, флаги, ошибки и т.п.).

### 5.3. Операции обмена рецептом

#### 5.3.1. Чтение рецепта
- Приложение должно уметь считать:
  - текущее количество шагов;
  - значения параметров для каждого шага по конфигурации;
  - состояние валидности рецепта (например, по флагу `recipe_invalid`).

#### 5.3.2. Запись рецепта
- Приложение должно уметь записывать изменения рецепта в ПЛК.

**Ключевое требование:**  
Запись и чтение **изменений** должны выполняться **атомарно на уровне логических операций**, без обязательной перезаписи всей области памяти.

- Нужна возможность:
  - изменять отдельные шаги и диапазоны шагов;
  - изменять отдельные параметры в шагах;
  - формировать минимально необходимый набор регистров для записи согласно конфигурации.
- При этом:
  - для каждой логической операции (например, «изменение значения параметра в одной ячейке рецепта») запись в ПЛК должна либо завершаться полностью успешно, либо откатываться/не учитываться (с точки зрения приложения и последующей работы с рецептом).
  - механизм атомарности на стороне ПЛК (staging-область, флаги «begin/end», версии) определяется отдельным протоколом взаимодействия, но требование на атомарность операции фиксируется в ТЗ.

### 5.4. Мониторинг исполнения
- Приложение должно считывать из ПЛК:
  - номер текущего шага;
  - состояние исполнения (идёт/остановлено/ошибка);
  - при необходимости — дополнительные статусы.
- Эти данные используются для:
  - подсветки текущего шага в таблице;
  - блокировки/ограничения редактирования при активном исполнении;
  - отображения информации в статус-баре.

### 5.5. Обработка ошибок связи
- При ошибках подключения/обмена:
  - отображать пользователю ясные сообщения о сбоях;
  - предотвращать переход приложения в неконсистентное состояние;
  - давать возможность повторить операцию без перезапуска приложения.

## 6. Интеграция с OPC и SCADA

### 6.1. Общая цель
- Обеспечить возможность работы с событиями и сигналами SCADA через OPC (UA/DA).
- OPC используется для обмена сигналами/событиями, не входящими в область чистого обмена по Modbus (например, высокоуровневые статусы, команды, события).

### 6.2. Поддерживаемые OPC-подключения
- OPC UA и/или OPC DA.
- Настройки:
  - URL/endpoint сервера;
  - базовые параметры безопасности (уточняются отдельно).

### 6.3. Использование OPC в приложении
Примеры возможного использования (подробная карта сигналов будет специфицирована отдельно):

- Получение состава выпадающих списков для действий (если есть).
- Получение сигналов о запуске/остановке процесса со стороны SCADA.
- Получение событий и тревог, связанных с рецептом или оборудованием, для отображения в приложении.
- Передача в SCADA информации о:
  - текущем рецепте;
  - состоянии редактирования/подготовки;
  - результатах валидации.
