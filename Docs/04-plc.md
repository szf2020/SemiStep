## 4.1. Общая концепция

- Конфигурация структуры рецепта хранится на стороне PC (YAML-файлы).
- Обмен данными выполняется по протоколу S7 (библиотека Sharp7).
- На стороне PLC выделяются отдельные датаблоки (DB) для разных типов данных.
- PLC не содержит логики валидации передачи — выступает как хранилище данных с флагом готовности.

---

## 4.2. Протокол и транспорт

| Параметр | Значение |
|----------|----------|
| Протокол | S7 (Siemens) |
| Библиотека | Sharp7 (чистый C# порт Snap7) |
| PLC | Siemens S7-1200 / S7-1500 |
| Порт | 102 (по умолчанию) |

### Характеристики транспорта

| Характеристика | Описание |
|----------------|----------|
| Гарантия доставки | TCP + S7 acknowledgement на каждый PDU |
| Целостность данных | Гарантирована на уровне TCP, CRC на стороне PLC не требуется |
| Атомарность | Один вызов DBWrite атомарен в пределах одного PDU; между вызовами — нет |
| PDU (S7-1200) | 240 байт (222 полезных) |
| PDU (S7-1500) | до 960 байт (942 полезных), negotiated при подключении |
| Разбиение больших запросов | Sharp7 автоматически разбивает на несколько PDU (PDU independence) |

### Настройки подключения

| Параметр | Описание |
|----------|----------|
| IP-адрес | Адрес ПЛК |
| Rack | Номер стойки |
| Slot | Номер слота CPU |
| DB номера | Номера датаблоков (задаются в конфигурации приложения) |

> **Примечание:** Номера DB статичны (задаются при компиляции проекта в TIA Portal). Хранятся в YAML-конфигурации приложения, отдельный Header DB для этого не требуется.

## 4.3. Структура данных в PLC

Данные разделены по датаблокам:

| Датаблок | Назначение |
|----------|------------|
| DB_MANAGING | Флаг готовности рецепта (committed) |
| DB_INT | Массив целочисленных данных рецепта |
| DB_FLOAT | Массив вещественных данных рецепта |
| DB_STRING | Массив строковых данных рецепта |
| DB_EXECUTION | Состояние выполнения рецепта |

### Managing Area (DB_MANAGING)

Минимальная область для координации записи:

- Флаг `committed` — признак что рецепт записан полностью и валиден
- Количество строк рецепта

> **Важно:** PLC не имеет возможности обнаружить факт внешней записи в DB. Коммуникационный процессор (CP) пишет в память прозрачно для пользовательской программы. Поэтому единственный способ сигнализации о завершении записи — явный флаг от PC.

---

## 4.4. Операции обмена

### 4.4.1. Чтение рецепта

PC считывает полное содержимое DB_INT, DB_FLOAT, DB_STRING. При объёме ~104 KB:

- S7-1200: ~5 секунд (PDU 240, ~470 round-trip)
- S7-1500: ~0.5 секунды (PDU 960, ~110 round-trip)

Sharp7 разбивает чтение на PDU автоматически.

### 4.4.2. Запись рецепта

**Ключевое требование:**

> Запись защищена флагом `committed`. PLC не использует рецепт, пока `committed = false`.

Механизм:

1. PC записывает `committed = false`
2. PC записывает данные в DB_INT, DB_FLOAT, DB_STRING (автоматическое разбиение на PDU)
3. PC записывает количество строк
4. PC записывает `committed = true`

Каждый отдельный WriteBytes подтверждается S7 протоколом. Если WriteBytes не выбросил исключение — данные гарантированно записаны.

### 4.4.3. Обработка ошибок

| Ошибка | Действие |
|--------|----------|
| Исключение при WriteBytes | Прервать, при реконнекте — перезаписать рецепт целиком |
| Обрыв связи на середине записи | `committed` остаётся `false`, при реконнекте — перезаписать целиком |
| Timeout подключения | Сообщение пользователю, повтор подключения |

> **Стратегия:** PC — source of truth. При любом сбое рецепт перезаписывается полностью из файла. Частичное дозаписывание не поддерживается и не требуется.

## 4.5. Мониторинг выполнения

Приложение периодически читает DB_EXECUTION:

| Данные | Назначение |
|--------|------------|
| `recipe_active` | Блокировка редактирования при выполнении |
| `actual_line` | Подсветка текущего шага в таблице |
| `step_current_time` | Расчёт оставшегося времени |
| `for_loop_count_1/2/3` | Отображение итераций вложенных циклов |

### Режимы UI

- **recipe_active = true** → режим мониторинга (редактирование заблокировано)
- **recipe_active = false** → режим редактирования

### Расчёт времени

```
LineTimeLeft = max(0, Длительность_шага − step_current_time)
TotalTimeLeft = max(0, Общая_длительность − (Время_начала_шага + step_current_time))
```

> **Примечание:** Polling DB_EXECUTION выполняется через отдельное S7-подключение или с синхронизацией доступа к общему подключению, чтобы не блокировать UI.

## 4.6. Обработка ошибок связи

При ошибках подключения/обмена:

- Отображать пользователю ясные сообщения о сбоях
- Предотвращать переход приложения в неконсистентное состояние
- Давать возможность повторить операцию без перезапуска приложения

### 4.6.1. Согласование при подключении (Reconnect Reconciliation)

При включении синхронизации приложение читает DB_MANAGING и выполняет согласование:

| Состояние PLC | Состояние PC | Действие |
|---------------|--------------|----------|
| `committed=false` | любое | Немедленно перезаписать рецепт из PC в PLC (без запроса) |
| `committed=true`, PLC = PC | совпадают | Нет действий |
| `committed=true`, PLC ≠ PC, PC пустой | PLC непустой | Загрузить рецепт из PLC в PC автоматически |
| `committed=true`, PLC ≠ PC, оба непустые | оба непустые | Показать диалог конфликта — пользователь выбирает источник |

> **Диалог конфликта:** предлагает сохранить локальную версию или загрузить версию из PLC. PC остаётся source of truth при выборе «сохранить локальную».

## 4.7. Управление синхронизацией

Синхронизация с PLC является **опциональной** и управляется пользователем через кнопку в статус-баре.

### 4.7.1. Включение синхронизации

При включении синхронизации («Sync: ON»):

1. Устанавливается S7-соединение с PLC
2. Выполняется согласование (см. 4.6.1)
3. При каждом успешном изменении рецепта запускается отложенная (1 сек) запись в PLC

### 4.7.2. Отключение синхронизации

При отключении синхронизации («Sync: OFF»):

- S7-соединение разрывается
- Мониторинг выполнения останавливается

### 4.7.3. Состояния синхронизации

В статус-баре отображается текущее состояние синхронизации:

| Состояние | Описание |
|-----------|----------|
| Idle | Синхронизация включена, ожидание изменений |
| Syncing | Запись рецепта в PLC выполняется |
| Synced | Рецепт успешно записан; время последней синхронизации |
| Out of sync | Рецепт невалиден и не будет записан |
| Failed | Ошибка при записи; текст ошибки отображается красным |
| Disconnected | Нет соединения с PLC |

### 4.7.4. Верификация записи

После каждой записи рецепта в PLC приложение считывает данные обратно и сравнивает с отправленными. При несоответствии — повтор до `MaxRetryAttempts` раз. При исчерпании попыток — статус `Failed`.
